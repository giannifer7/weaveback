= Weaveback CLI spec
:toc: left

This document is the macro-driven source of truth for the `weaveback` CLI
surface. It owns two projections:

- [cli_generated.rs](/home/g4/_prj/weaveback/crates/weaveback/src/cli_generated.rs)
- [cli.adoc](/home/g4/_prj/weaveback/docs/cli.adoc)

The point is not cleverness for its own sake. The point is to make one bounded
surface adopt one bounded abstraction uniformly:

- command-line option declarations
- parser-facing Rust fields
- CLI reference tables

If this layer is worthwhile, it should lower drift across those views. If it
does not, it should be removed.

== What This File Is Doing

The design is simple, but the current file hid it too aggressively.

There are three layers:

1. option-family macros in [macros.adoc](/home/g4/_prj/weaveback/cli-spec/macros.adoc)
2. projection blocks in this file
3. generated targets:
   - [cli_generated.rs](/home/g4/_prj/weaveback/crates/weaveback/src/cli_generated.rs)
   - [cli.adoc](/home/g4/_prj/weaveback/docs/cli.adoc)

The important point is that the option families are not supposed to write Rust
or docs directly in one opaque burst. They should contribute rows to named
chunks, and the generated file skeleton should stay readable.

=== Mental model

An option family like `%%search_command_options()` is just a reusable list of
option declarations. We project that family more than once:

* into Rust fields / enum members for clap
* into AsciiDoc table rows for the CLI reference

That means the same option metadata drives both the parser and the docs.

=== Reading order

If you want to understand the file without reading every macro expansion:

1. read the generated Rust skeleton below
2. note the named chunks it pulls in
3. look at the projection blocks that fill those chunks
4. only then look at the reusable option-family definitions in [macros.adoc](/home/g4/_prj/weaveback/cli-spec/macros.adoc)

=== Example: one option, two projections

The shape is:

[source]
----
%%emit_option(
field = limit,
rust_type = usize,
arg_attr = %{long, default_value = "10"%},
doc_block = %{    /// Maximum number of results to show%},
doc_flag = %{`--limit <N>`%},
doc_default = %{`10`%},
doc_descr = %{Maximum number of results to show.%}
)
----

From that one declaration we want:

* a Rust field with `#[arg(...)]`
* a doc-table row

The accumulating chunk approach makes that explicit instead of forcing the
reader to mentally inline every macro call.

    Where {

    },

    Tangle {

    },

    },
    /// Run cargo with JSON diagnostics and add source-of-truth attribution
    Cargo {
// <<cli-rust-cargo-options>>
    },

    Tag {

    },

    Search {

== `weaveback` — combined tool

Runs macro expansion and chunk extraction as one command.

[source,bash]
----
weaveback [OPTIONS] <INPUTS>...      # explicit files
weaveback [OPTIONS] --dir <DIR>      # auto-discover driver files
----

=== Options

[cols="3,1,4",options="header"]
|===
| Flag | Default | Description
<<@tight cli-doc-main-rows>>
|===

=== Directory mode

`--dir` scans a directory tree recursively for files matching `--ext`,
determines which are _drivers_ (top-level files) vs _fragments_ (included by
another file), and processes each driver. No changes needed when new files are
added.

[source,bash]
----
weaveback --dir src --include . --gen src
weaveback --dir src --ext adoc --include . --gen src
weaveback --dir src --ext md --ext adoc --include . --gen src
----

`--dump-expanded` is the first thing to check when a chunk can't be found or
expands unexpectedly — it prints the macro-expanded intermediate text that
weaveback-tangle receives.

=== Build-system integration

`--depfile` and `--stamp` together let a single build rule cover an entire
directory of literate sources:

[source,meson]
----
custom_target('gen',
  output  : ['gen.stamp'],
  depfile : 'gen.d',
  command : [weaveback,
             '--dir',     meson.current_source_dir() / 'src',
             '--ext',     'adoc',
             '--include', meson.current_source_dir(),
             '--stamp',   '@OUTPUT0@',
             '--depfile', '@DEPFILE@'],
)
----

NOTE: List only the stamp in `output`, never the `.d` file.
Ninja consumes the depfile internally; declaring it as an output makes ninja
think it is always missing and reruns the target on every build.

'''

==== `cargo`

Runs `cargo` with JSON diagnostics enabled and augments compiler messages with
source-of-truth attribution when a span maps back to a generated file.

[source,bash]
----
weaveback cargo check
weaveback cargo clippy --all-targets -- -D warnings
----

==== `apply-back`

When you edit a file in `gen/` directly, the next `weaveback` run refuses to
overwrite it (`ModifiedExternally`). `apply-back` closes the loop: it diffs
the modified gen/ file against the stored baseline, traces each changed line
back through noweb and macro provenance, then tries to reconstruct the best
literate-source edit that reproduces the desired generated result.

[source,bash]
----
weaveback apply-back                     # process all modified gen/ files
weaveback apply-back src/foo.c           # process one specific file
weaveback apply-back --dry-run           # show patches without writing
weaveback --gen path/to/gen apply-back   # use non-default gen/ directory
----

For literal and simple noweb-mapped edits, the patch can be applied directly.
For macro-generated regions, weaveback now does bounded candidate search plus
oracle verification. Size-changing hunks are still reported for manual
resolution instead of being guessed silently.

==== `serve`

Starts a local HTTP server at `http://127.0.0.1:<port>/` that serves
`docs/html/`, pushes live-reload events to connected browsers when HTML files
change, and provides the inline chunk editor for literate source files.

[source,bash]
----
weaveback serve                          # default port 7779
weaveback serve --port 8080             # custom port
weaveback serve --html /other/dir       # serve a different directory
weaveback serve --watch                 # rebuild docs on source changes
----

[cols="3,2,4",options="header"]
|===
| Flag | Default | Description
<<@tight cli-doc-serve-rows>>
|===

The tangle-oracle flags must match the delimiters used by the project being
served. They are only needed if you use non-default chunk syntax and intend
to use the `/__apply` inline editor endpoint.

=== AI Backend Modes

`claude-cli` (default):: Shells out to `claude -p --output-format stream-json`,
  using the authenticated Claude Code session already on the machine — no extra
  credentials required.
`anthropic`:: Calls the Anthropic Messages API directly. Requires the
  `ANTHROPIC_API_KEY` environment variable.
`gemini`:: Calls the Google Gemini API. Requires the `GOOGLE_API_KEY`
  environment variable.
`ollama`:: Calls a local Ollama instance. Uses `--ai-endpoint` (default:
  `http://localhost:11434`) and `--ai-model` (default: `llama3`).
`openai`:: Calls an OpenAI-compatible API. Requires `OPENAI_API_KEY` (unless
  using a local provider) and `--ai-endpoint`.

See link:architecture.adoc#_live_documentation_server_weaveback_serve[architecture.adoc]
for the full design, HTTP endpoint reference, and inline editor documentation.
