= Weaveback Macro Language Reference
:toc: left
:toc-title: Contents
:source-highlighter: syntect
:icons: font

This document is the normative specification of the weaveback macro language.

== Overview

The macro language is a **strict, eager, string-valued expansion system**.
Every macro call reduces to a string.  There are no first-class booleans,
numbers, or lists outside of the Python escape hatch.

The language has two concerns that are deliberately kept separate:

* **Macro definitions** — `%def`, `%redef`, `%pydef`, `%alias`
* **Output generation** — calling defined macros and splicing their results
  into the text stream

A weaveback run processes one or more _driver_ files.  Each file is parsed
into an AST and then _evaluated_: text nodes pass through unchanged, macro
calls are dispatched and their results are spliced in.

=== Sigil

The default sigil is `%`.  Every syntactic construct in the language begins
with the sigil.  The sigil itself can be changed per-run with `--sigil`.
A doubled sigil (`%%`) is a literal single sigil character; the macro
expander never processes it as a call.

[source]
----
%foo(x)          → macro call
%%foo(x)         → literal text: %foo(x)
----

The `--sigil ^` convention is the intended workaround for documents that
contain many literal percent signs (e.g., CSS, shell scripts).

== Syntax

=== Token types

[cols="1,3"]
|===
| Token | Description

| `%name(…)` | Macro call.  `name` is an identifier `[A-Za-z_][A-Za-z0-9_]*`.
| `%(name)` | Variable reference.
| `%{ … %}` / `%tag{ … %tag}` | Macro-aware block (quotes commas and parentheses).
| `%[ … %]` / `%tag[ … %tag]` | Opaque verbatim block (disables macro parsing inside).
| `%%` | Escaped sigil — expands to a single literal sigil character.
| `%// …` | Line comment — discarded.
| `%/* … %*/` | Block comment — discarded.
|===

Everything else is literal text.

=== Anonymous blocks `%{ … %}`

A block is syntactically a single macro argument.  Its primary use is to
include commas or closing parentheses literally inside an argument that would
otherwise be split:

[source]
----
%def(greet, name, %{Hello, %(name)!%})
----

Without the block, the comma after `Hello` would split the argument list into
`Hello` and ` %(name)!`, giving `greet` four arguments instead of three.

Blocks have **no scope of their own**.  They are evaluated in the enclosing
context.

Use `%{%}` to pass an explicit empty string argument — the parser strips
whitespace-only arguments, so a comma with only spaces is not a reliable way
to pass empty.

Tagged blocks `%tag{ … %tag}` behave the same way, but use an explicit tag
for readability and nesting discipline.

=== Verbatim blocks `%[ … %]` and `%tag[ … %tag]`

Verbatim blocks are opaque to macro parsing:

* no macro calls are expanded
* no `%(var)` interpolation happens
* comment syntax is not recognised
* nesting is allowed, including tagged nesting

They are the general-purpose “treat this region literally” mechanism, useful
for embedded scripts, regexes, templates, or any local `--nomacro` behaviour.

[source]
----
%pydef(greet, name, %[ "hello " + name %])
----

=== Identifiers

An identifier is `[A-Za-z_][A-Za-z0-9_]*`.  Hyphens are *not* identifier
characters; `foo-bar` parses as `foo` (identifier), `-bar` (text).  This is a
common source of confusion when porting shell variable names or CSS class names.

== Evaluation model

=== Strict, eager expansion

All arguments to a macro call are **fully expanded in the caller's current
scope before the macro body runs**.  There is no lazy evaluation for
user-defined macros.  Consider:

[source]
----
%def(emit_counter, %(counter))
%def(id, x, before=%emit_counter() arg=%(x) after=%emit_counter())
%set(counter, 0)
%id(%set(counter, 1)%(counter))
----

Output: `before=0 arg=1 after=0`.  The argument is fully expanded in the
**caller scope** before `id` is entered, then the resulting string is bound to
`x` in the callee frame.

The one exception is `%if`, which evaluates its condition first and then
evaluates only the selected branch.

=== Scope stack

The evaluator maintains a `Vec<ScopeFrame>`.  Each frame holds:

* `variables: HashMap<String, String>` — string bindings
* `macros: HashMap<String, MacroDefinition>` — macro bindings

Frame 0 is the global frame; it is never popped.  Each macro call pushes a
fresh empty frame, binds parameters into it, runs the body, and pops on return.

**Lookup** walks from the top of the stack to frame 0 and returns the first
match.  A binding in an inner scope shadows the same name in an outer scope.

**Assignment** (`%set`, parameter binding) always targets the **current top
frame**.  There is no way to assign to an outer scope directly; use `%export`
for that.

=== Call dispatch

When `%foo(…)` is encountered:

. Look up `foo` in the **builtin registry** first.  Builtin names are
  permanently reserved; user macros cannot shadow them.
. If not a builtin, walk the scope stack for a `MacroDefinition`.
. If neither is found, return `EvalError::UndefinedMacro`.

Attempting to define a user macro with a builtin name (`%def(set, …)`) is
immediately rejected with `EvalError::InvalidUsage`.  This applies to `%def`,
`%redef`, `%alias` and `%pydef`.

=== Argument binding

Arguments are evaluated in the **caller's scope** before the callee frame is
pushed.

Given a call `%foo(a, b, key = c)` and a definition `%def(foo, p, q, r, body)`:

. *Validate order*: Positional args must precede named args.  A positional
  arg after a named arg is `EvalError::InvalidUsage`.
. *Positional binding*: Arg 0 → `p`, arg 1 → `q`.
. *Named binding*: `key` is looked up in `{p, q, r}`.  Unknown named arg →
  `EvalError::InvalidUsage`.
. *Duplicate binding*: Same param bound both positionally and by name →
  `EvalError::InvalidUsage`.
. *Extra positional args*: `EvalError::InvalidUsage` — too many positional
  arguments is a bug, not a tolerated shape.
. *Effectful builtins in arguments*: `%set(...)` in argument position is
  `EvalError::InvalidUsage`. Arguments are values, not assignment sites.
. *Missing params*: Unbound params are `UnboundParameter` by default.
  `--no-strict-params` / `EvalConfig.strict_unbound_params = false` restores
  the old empty-string fallback.

== Variables

=== `%(name)` — reference

Looks up `name` in the scope stack (top-to-bottom).  If not found, expands to
empty string (no error).  A typo in a variable name produces blank output with
no diagnostic.

=== `%set(name, value)` — assignment

Evaluates `value` and stores it as `name` in the current top frame.

* `name` must be a single identifier.
* Records the definition location in the source-map database.

---

== Macro definitions

=== `%def(name, [params…,] body)`

Defines a **constant** text-substitution macro in the current frame.

[cols="1,3"]
|===
| Field | Rule

| `name` | Single identifier. Must not be a builtin name. Errors if the name already exists in the current frame as either constant or rebindable.
| `params` | Zero or more identifiers, comma-separated.  Duplicates are an error.  A trailing comma is ignored.
| `body` | Last argument.  Typically an anonymous block `%{ … %}` when the body contains commas or parentheses.
|===

The macro body is stored as an `Arc<ASTNode>` — cloning a macro definition is
`O(1)` (no deep copy of the body).

[[macro-redefinition]]
=== `%redef(name, [params…,] body)`

Defines or replaces a **rebindable** macro in the current frame.

[cols="1,3"]
|===
| Field | Rule

| `name` | Single identifier. Must not be a builtin name. Errors if the name already exists in the current frame as a constant binding. If the name already exists as rebindable in the current frame, it is replaced.
| `params` | Zero or more identifiers, comma-separated. Duplicates are an error. A trailing comma is ignored.
| `body` | Last argument. Typically an anonymous block `%{ … %}` when the body contains commas or parentheses.
|===

=== Constant vs. rebindable names

The definition model is intentionally explicit:

* `%def` creates a constant binding
* `%redef` creates or replaces a rebindable binding

That gives one simple invariant:

* constant names stay constant
* rebindable names are explicitly marked

Use `%redef` for deliberate X-macro or multi-pass rebinding patterns.

=== Rebinding patterns

X-macro pattern::
A schema is defined once; successive passes rebind the `X` visitor to map
that schema into different outputs.

Context-dependent meaning::
Macro identifiers have no fixed meaning.  Their expansion is determined by the
current pass, allowing the same source to be interpreted multiple ways.

[source]
----
%def(FIELDS, %{
  %X(name, string)
  %X(age, int)
%})

%redef(X, name, type, %(name): %(type),)          <- pass 1: struct fields
%FIELDS()

%redef(X, name, type, new_%(name): impl Into<%(type)>)  <- pass 2: ctor params
%FIELDS()
----

Use `%importas` when combining independent libraries to prevent accidental
name collision.

=== `%pydef(name, [params…,] body)`

Same argument structure as `%def`. At call time, the body is **first
macro-expanded** (substituting `%(param)` references), then executed as Python
(monty) source code. The return value of the script is the macro's expansion.

Declared parameters are injected as Python variables. Ordinary `%set`
variables are **not** automatically injected into the Python scope; if Python
code needs their values, they must be interpolated during macro expansion
before execution.

Because the body undergoes macro expansion before execution, any literal `%`
sigil in the script source must either be doubled (`%%`) or placed inside a
verbatim block so the script sees it literally.

=== `%alias(new_name, source_name [, key = val, …])` — Macro aliasing

Creates a new macro definition that is a copy of `source_name` at the moment
`%alias` is called.  The copy shares the body `Arc` (`O(1)` clone).

**Snapshot semantics**: Later redefinition of `source_name` does not affect
the alias.

**Free-variable pre-binding** (optional named args): Each `key = val` pair is
evaluated at alias time and stored as a frozen binding.  Whenever the alias is
called, those bindings are installed in the callee's scope before parameter
binding.  This is the **only** capture mechanism in the language:

[source]
----
%def(render_row, msg, chunk_name, %{| %(msg) |
%})

%alias(emit_tangle_row, render_row, chunk_name = cli-doc-tangle-rows)
%emit_tangle_row(my option description)
----

The `chunk_name` free variable is pinned to `cli-doc-tangle-rows` for all
calls to `emit_tangle_row`.

A frozen binding on a formal parameter name is shadowed by the call-site
value (parameter wins over frozen default).

=== `%export(name)` — Propagate to parent scope

Copies the binding for `name` one level up the scope stack (to the parent
frame).

* For both variables and macros, the binding is copied **as-is** — no
  automatic free-variable freezing occurs.
* At global scope (no parent frame), `%export` is a no-op and emits a
  non-fatal warning.

To create a macro that carries frozen bindings into the parent scope, use
`%alias` to create the frozen copy first, then `%export` it.

== Boolean model

The macro language has no boolean type.  The convention used by all builtins:

* empty string → false
* any non-empty string → true

Predicates return `1` (truthy) or empty (falsy).

Note: the parser strips leading whitespace from unquoted arguments.  An
unquoted whitespace-only argument (`%foo(   )`) arrives as empty string and
is falsy.  Content inside `%{ … %}` blocks is preserved verbatim, so
`%foo(%{   %})` receives three spaces, which is non-empty and truthy.

== Control flow

=== `%if(cond, then [, else])` — Conditional

Evaluates `cond`.  If the result is non-empty, evaluates and returns `then`;
otherwise evaluates and returns `else` (or empty if `else` is omitted).

The **non-selected branch is not evaluated** — `%if` is the one builtin that
is genuinely lazy in its branch arguments.

`%if()` with no arguments is a non-fatal warning (the call always returns
empty and is almost certainly a mistake).

=== `%eq(a, b)` — Equality predicate

Evaluates both arguments, compares byte-exact.  Returns `1` if equal, empty
if not.

=== `%neq(a, b)` — Inequality predicate

Returns `1` if `a != b`, empty if equal.  `%eq` and `%neq` are strict
inverses.

=== `%not([x])` — Logical negation

Accepts zero or one argument.  Returns `1` if the argument is empty (or absent),
empty if the argument is non-empty.

`%not()` with no argument: treats the absent argument as empty, so returns `1`.
`%not(a, b)` is `EvalError::InvalidUsage`.

=== `%equal(a, b)` — Legacy equality

Returns `a` if `a == b`, empty if not.  Retained for backward compatibility.
Prefer `%eq` in new code.

=== `%eval(name, args…)` — Dynamic dispatch

Evaluates the first argument to obtain a macro name, then calls that macro with
the remaining arguments.  Enables data-driven dispatch.

[source]
----
%def(render_html, x, <b>%(x)</b>)
%def(render_md, x, **%(x)**)
%eval(render_%(fmt), hello)   ← fmt = "html" or "md"
----

=== `%here(name, args…)` — Source-file patching

Calls the macro `name(args…)`, writes the result into the **source file itself**
immediately after the `%here(...)` call, then stops evaluation (`early_exit`).

On the next run, the `%here(...)` call has been patched to `%%here(...)`,
which expands to literal `%here(...)` and is never dispatched again.  This
makes `%here` idempotent.

Caveats:

* Only one `%here` fires per run (the first encountered).  Multiple `%here`
  calls in one file are effectively a bug.
* The source file must be writable.
* `%here` is a source-rewrite operation, not a normal expression-forming macro.
  Generated code should not contain it.

Once `%here` sets `early_exit`, all subsequent `evaluate()` calls return empty
immediately.  This is not an error condition.

== Diagnostics

=== Errors

[cols="1,3"]
|===
| Error | Trigger

| `UndefinedMacro(name)` | Macro `%name` not found in scope or builtins
| `InvalidUsage(msg)` | Wrong arg count; invalid identifier; positional-after-named;
  duplicate binding; unknown named arg; extra positional args;
  attempt to define a builtin name; invalid `%def` / `%redef` rebinding
| `Runtime(msg)` | Recursion limit exceeded; script engine error; `%here` I/O failure
| `ParseError(msg)` | Lexer or parser failure in a file being evaluated
| `IncludeNotFound(path)` | `%include`/`%import`/`%importas` path not resolved
| `CircularInclude(path)` | File is already on the include stack
| `IoError(e)` | Filesystem error during file inclusion
|===

=== Warnings (non-fatal)

Warnings are accumulated in the evaluator and drained by the caller via
`take_warnings()`.  They do not abort evaluation.

[cols="1,3"]
|===
| Warning | Trigger

| `%if() with no arguments` | `%if()` called with zero arguments; always returns empty
| `%export at global scope` | `%export` called when there is no parent frame; the call is a no-op
|===

== String transformations

All case-conversion builtins apply **word-aware splitting** before
transformation.  Word boundaries are:

* `_`, `-`, space
* CamelCase transition (lower→upper)
* Acronym boundary (upper→upper→lower)
* Digit transition

[cols="1,2,1"]
|===
| Builtin | Transformation | Example

| `%capitalize(s)` | Upper-case first character | `hello` → `Hello`
| `%decapitalize(s)` | Lower-case first character | `Hello` → `hello`
| `%to_snake_case(s)` | Lower, underscore-separated | `FooBar` → `foo_bar`
| `%to_camel_case(s)` | Lower first word, upper rest | `foo_bar` → `fooBar`
| `%to_pascal_case(s)` | Upper every word | `foo_bar` → `FooBar`
| `%to_screaming_case(s)` | Upper, underscore-separated | `FooBar` → `FOO_BAR`
| `%convert_case(s, style)` | Dispatch by `style` string | —
|===

All return empty string if input is empty.

== File inclusion

=== `%include(path)` — Include and evaluate

Resolves `path` against the include-path list.  Reads and evaluates the file
inline; its output is spliced at the call site.  Macro definitions made in the
included file are visible to the caller.

Circular includes are detected at runtime and return `EvalError::CircularInclude`.

An included file runs in the caller's current scope frame.  `%set` and `%def`
inside the included file land in that frame.  At top level this means global;
inside a macro body, definitions are local and disappear on pop.

=== `%import(path)` — Include without output

Like `%include` but discards the text output.  Side effects (macro definitions,
`%set`) are preserved.

[source]
----
%import(macros.adoc)   ← load definitions, produce no output
%render_doc(...)
----

=== `%importas(prefix, path)` — Namespaced import

Runs `path` like `%import` (text output discarded, side effects kept) and
additionally registers every macro **newly defined by the import** under a
`prefix_name` alias.  The originals remain in scope so internal
cross-references within the imported file continue to work.

[source]
----
%importas(mylib, lib.adoc)
%mylib_greet(world)   ← "Hello world!"
%greet(world)         ← also works (original name remains)
----

`prefix` must be a valid identifier.  Use `%importas` when combining
independent libraries to prevent accidental name collision.

== Python integration

=== Persistent stores

The Python store (`py_store: HashMap<String, String>`) survives across macro
calls and across `%def`/`%pydef` definitions for the lifetime of the evaluator
session. It is **not** part of the scope stack and is **not** popped with
frames.

[cols="1,1,3"]
|===
| Builtin | Store | Description

| `%pyset(k, v)` | Python | Store string `v` as key `k`
| `%pyget(k)` | Python | Return value for key `k`; empty if missing
|===

Script stores bypass all scope-stack discipline.  They are powerful for
accumulation patterns but make
behaviour order-dependent.  Prefer local `%set` where possible.

=== `%env(NAME)` — Environment variable

Reads `NAME` from the process environment.  Requires `--allow-env`; returns
empty string if not found or if `--allow-env` is not set.

---

== Recursion limit

The evaluator tracks call depth.  Exceeding `MAX_RECURSION_DEPTH` (defined in
`weaveback_core`) produces `EvalError::Runtime`.  There is no tail-call
optimisation; every call frame occupies stack space.

---

== Configuration

`EvalConfig` is constructed by the caller and passed to the `Evaluator`.

[source,rust]
----
pub struct EvalConfig {
    pub sigil: char,                  // default: '%'
    pub include_paths: Vec<PathBuf>,  // default: ["."]
    pub discovery_mode: bool,         // default: false
    pub allow_env: bool,              // default: false
    pub strict_undefined_vars: bool,  // default: true
    pub strict_unbound_params: bool,  // default: true
}
----

**Discovery mode**: `%include`/`%import` record the resolved path in
`discovered_includes` without evaluating.  Used to collect the dependency
graph for incremental builds.

[[open-items]]
== Open items

The following behaviours are known limitations and candidates for future work:

. **Undefined variable strictness** — `%(typo)` is `UndefinedVariable` by
  default.  `--no-strict-vars` / `EvalConfig.strict_undefined_vars = false`
  restores the old empty-string behaviour.  `%defined(name)` / `%default(x,
  fallback)` style helpers would still improve explicitness.

. **Missing (unbound) parameter** — unbound formal parameters are
  `UnboundParameter` by default.  `--no-strict-params` /
  `EvalConfig.strict_unbound_params = false` restores the old empty-string
  fallback.  Explicit optional parameters with inline defaults (`%def(fmt,
  level, tag, msg = "(none)", body)`) are a possible future addition.

. **`%(name)` variable reference vs. `%name()` call ambiguity** — `%set` and
  `%(name)` operate on different namespaces (variables vs. macros).  This is
  intentional but occasionally surprising.

. **`%here` — single-fire** — only the first `%here` in a file executes per
  run; multiple uses are silently broken.  A hard error on the second `%here`
  in a file would be better.

. **`MAX_RECURSION_DEPTH` not user-configurable** — value is a compiled-in
  constant, not surfaced in config or error messages.

. **No builtin string operations** — no substring, trim, length, or join;
  these require Python or multi-step `%def` chains.

. **Discovery mode API** — currently a flag on `EvalConfig`; ideally a
  separate `discover_dependencies()` entry point so the same source text
  cannot accidentally be evaluated in two radically different ways depending
  on a hidden flag.
